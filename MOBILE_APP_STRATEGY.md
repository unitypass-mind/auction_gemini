## 전략적 배경

### 왜 모바일 앱인가?

#### 웹 서비스의 한계
1. **경쟁 포화**: 부동산 경매 정보 웹사이트 다수 존재
   - 지지옥션, 태인옥션, 굿옥션 등 기존 플레이어
   - 낮은 진입 장벽 (누구나 웹사이트 개설 가능)
   - SEO 경쟁 치열 (상위 노출 어려움)

2. **수익화 어려움**
   - 광고 수익: 낮은 CPM (1,000회당 $1-3)
   - 구독 모델: 웹에서 결제 전환율 낮음 (1-2%)
   - 경쟁사 대부분 무료 제공

3. **사용자 락인(Lock-in) 부족**
   - 브라우저 북마크 의존
   - 알림(Push notification) 제한적
   - 앱 대비 재방문율 낮음

#### 모바일 앱의 강점
1. **진입 장벽 높음**
   - 앱 개발 전문성 필요
   - 앱스토어 심사 통과 필요
   - 초기 투자 비용 상대적으로 높음
   → **경쟁사 수 제한적**

2. **사용자 락인 효과**
   - 홈 화면 아이콘 (항상 보임)
   - 푸시 알림 (사용자 재방문 유도)
   - 앱 삭제 심리적 허들 높음

3. **수익화 유리**
   - 인앱 결제 전환율 높음 (5-10%)
   - 구독 모델 적합 (월간/연간)
   - 앱스토어 신뢰도 높음 (결제 안전)

4. **검색 발견성 (Discoverability)**
   - 웹 SEO vs 앱스토어 ASO 비교
   - 앱스토어 카테고리 검색 우위
   - "부동산 경매" 키워드 검색 시 앱 노출

### 전략적 결정
- **1단계 (3-4월)**: 모바일 앱 단독 출시 (웹 중단)
- **2단계 (5-6월)**: 앱 기능 고도화 (AI 고급 분석, 알림)
- **3단계 (7월~)**: 웹 관리자 대시보드 추가 (옵션)

## 시장 분석

### 타겟 고객

#### 주요 타겟 (Primary)
1. **경매 초보 투자자** (25-45세)
   - 부동산 경매 관심 있으나 정보 부족
   - 낙찰가 예측으로 진입 장벽 낮춤
   - **시장 규모**: 연 10만명 이상 신규 경매 참여자

2. **재테크 관심층** (30-50세)
   - 주식, 코인 외 대체 투자 관심
   - 모바일 앱 사용 친숙
   - **시장 규모**: 모바일 재테크 앱 사용자 500만명+

#### 부차 타겟 (Secondary)
3. **전문 경매 투자자**
   - 데이터 기반 의사결정 선호
   - 다수 물건 동시 분석 필요
   - **시장 규모**: 연간 경매 낙찰 건수 15만건

### 시장 규모 추정

```
한국 부동산 경매 시장:
- 연간 경매 물건: 약 25만 건
- 연간 낙찰 건수: 약 15만 건 (낙찰률 60%)
- 시장 규모: 약 30조원 (평균 낙찰가 2억)

잠재 앱 사용자:
- 경매 관심층: 100만명
- 초기 목표: 5,000 다운로드 (1개월)
- 1년 목표: 50,000 다운로드
- 유료 전환율: 5% → 2,500명
```

### PWA Wrapper 아키텍처

#### 왜 PWA Wrapper인가?

**장점**:
1. **개발 속도**: 기존 웹 코드 재사용 (80%)
2. **유지보수**: 서버 업데이트만으로 앱 기능 추가
3. **비용**: Native 대비 1/5 수준
4. **배포**: 앱스토어 심사 1회, 이후 서버만 업데이트

**단점**:
1. **성능**: Native 대비 5-10% 느림 (무시할 수준)
2. **앱스토어 정책**: 일부 제한 (우회 가능)
3. **고급 기능**: 카메라, GPS 등 제한적 (불필요)

#### 핵심 구현
```kotlin
// MainActivity.kt (간략화)
class MainActivity : AppCompatActivity() {
    private lateinit var webView: WebView

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        webView = WebView(this)
        webView.settings.apply {
            javaScriptEnabled = true
            domStorageEnabled = true
            cacheMode = WebSettings.LOAD_DEFAULT
        }

        webView.loadUrl("https://api.auction-ai.com")

        // FCM 토큰 전송
        FirebaseMessaging.getInstance().token.addOnCompleteListener { task ->
            if (task.isSuccessful) {
                val token = task.result
                sendTokenToServer(token)
            }
        }
    }
}
```

## 성공 지표 (KPI)

### 앱 성과 지표

#### 1. 다운로드 지표
| 기간 | 목표 다운로드 | 실제 | 달성률 |
|------|---------------|------|--------|
| 1개월 (4월) | 1,000 | - | - |
| 3개월 (6월) | 5,000 | - | - |
| 6개월 (9월) | 10,000 | - | - |
| 1년 (2027/3) | 50,000 | - | - |

#### 2. 사용자 활성도
- **DAU (Daily Active Users)**: 일일 활성 사용자
  - 목표: 전체 사용자의 10%
- **WAU (Weekly Active Users)**: 주간 활성 사용자
  - 목표: 전체 사용자의 30%
- **MAU (Monthly Active Users)**: 월간 활성 사용자
  - 목표: 전체 사용자의 60%

#### 3. 전환율
- **무료→유료 전환율**: 3-5% 목표
- **이탈률 (Churn Rate)**: 월 5% 이하
- **재구매율**: 90% 이상 (구독 갱신)

#### 4. 수익 지표
- **ARPU (Average Revenue Per User)**: 사용자당 평균 매출
  - 목표: 월 1,000원 (전체 사용자 기준)
- **LTV (Lifetime Value)**: 고객 생애 가치
  - 목표: 120,000원 (1년 구독 기준)
- **CAC (Customer Acquisition Cost)**: 고객 획득 비용
  - 목표: 30,000원 이하 (LTV/CAC > 3)

### 기술 지표

#### 1. 성능
- **API 응답 시간**: 평균 < 500ms, P95 < 1,000ms
- **예측 응답 시간**: 평균 < 2초, P95 < 3초
- **앱 시작 시간**: < 2초

#### 2. 안정성
- **서버 가동률 (Uptime)**: 99.5% 이상
- **에러율**: < 1%
- **크래시율**: < 0.5%

#### 3. AI 정확도
- **예측 오차율**: 평균 10% 이하
  - 실제 낙찰가 대비 ±10% 이내 적중률 70%
- **권리분석 만족도**: 평균 4.0/5.0 이상

## 위험 관리

### 기술적 위험

#### 1. API 의존성 리스크
**위험**: Google Gemini API 장애 또는 가격 인상
**완화 전략**:
- Gemini API 없이도 기본 예측 기능 동작 (fallback)
- API 비용 모니터링 (월 예산 설정)
- 대체 API 사전 조사 (OpenAI GPT-4, Anthropic Claude)

#### 2. 서버 장애
**위험**: NCP 서버 다운 → 앱 사용 불가
**완화 전략**:
- NCP 이중화 (추후 검토)
- 백업 서버 준비 (AWS 또는 GCP)
- 로컬 캐싱 (최근 예측 결과 앱에 저장)

#### 3. 데이터 부족
**위험**: 학습 데이터 부족 → AI 정확도 하락
**완화 전략**:
- 지속적인 낙찰 데이터 수집 (크롤링)
- 사용자 피드백 수집 ("실제 낙찰가 알려주기")
- 모델 재학습 주기: 월 1회

### 비즈니스 위험

#### 1. 사용자 확보 실패
**위험**: 목표 다운로드 미달
**완화 전략**:
- 초기 마케팅 예산 확보 (월 100만원)
- ASO 지속 최적화
- 무료 체험 확대 (3회 → 5회)

#### 2. 유료 전환율 저조
**위험**: 무료 사용자만 증가, 매출 부족
**완화 전략**:
- 무료 기능 제한 강화 (월 3회 → 월 1회)
- 유료 가치 강조 (예: "월 1만원으로 수백만원 절약")
- A/B 테스트 (가격, 기능 조합)

#### 3. 경쟁사 등장
**위험**: 유사 앱 출시 (대기업 진입)
**완화 전략**:
- 선점 효과 극대화 (빠른 출시)
- 데이터 축적 (경쟁사가 따라잡기 어려움)
